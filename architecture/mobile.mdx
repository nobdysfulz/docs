PWRU implements a comprehensive mobile experience using viewport detection, dedicated mobile page variants, and responsive utilities.

## Viewport Detection

Every major page uses the same pattern to detect mobile viewports:

```jsx
const [isMobile, setIsMobile] = useState(false);

useEffect(() => {
  const checkMobile = () => setIsMobile(window.innerWidth < 768);
  checkMobile();
  window.addEventListener('resize', checkMobile);
  return () => window.removeEventListener('resize', checkMobile);
}, []);

if (isMobile) {
  return <MobilePageVariant />;
}
```

**Breakpoint**: 768px is the consistent mobile/desktop threshold.

Each major page has a dedicated mobile component:

| Desktop Page | Mobile Variant | File |

| `ToDo` | `MobileHomePage` | `MobileHome.jsx` |

| `Crm` | `MobileContactsPage` | `MobileContacts.jsx` |
| `ContentStudio` | `MobileContentStudioPage` | `MobileContentStudio.jsx` |

| `Settings` | `MobileSettingsPage` | `MobileSettings.jsx` |

Mobile variants are **not separate routes** but are conditionally rendered within the same page component based on viewport width.

## Mobile Navigation

### Bottom Navigation Bar

The mobile layout replaces the desktop sidebar with a fixed bottom navigation bar:
- 5 primary navigation items with icons and labels
- Fixed height of 64px plus safe-area-inset
- Active tab highlighting
- Centered icon + label layout

### No Top Header

The desktop `TopHeader` is hidden on mobile. Search and profile access are handled differently in each mobile page variant.

## Touch Interactions

### Pull-to-Refresh

Several mobile pages implement pull-to-refresh using a custom `PullToRefresh` component:
- Touch gesture detection (pull down from top)
- Loading spinner during refresh
- Triggers React Query refetch
- Available on: ToDo, CRM, Goals

### Swipe Gestures

The contextual sidebar supports swipe-to-dismiss on mobile:
- Swipe right to close the sidebar
- Touch the overlay backdrop to close

### Touch Targets

Mobile components use larger touch targets:
- Minimum 48px tap area for interactive elements
- Increased padding on buttons and list items
- Larger font sizes for improved readability

## Responsive Layout Utilities

### Tailwind Breakpoint Classes

The application uses Tailwind's responsive prefixes:

| Pattern | Behavior |
|---------|----------|
| `hidden md:block` | Hidden on mobile, visible on desktop |
| `block md:hidden` | Visible on mobile, hidden on desktop |
| `w-full md:w-96` | Full-width mobile, fixed-width desktop |
| `px-4 md:px-6` | Tighter padding on mobile |
| `text-sm md:text-base` | Smaller text on mobile |
| `grid-cols-1 md:grid-cols-2` | Single column mobile, multi-column desktop |

### Sidebar Responsiveness

The `ContextualSidebar` adapts its behavior by breakpoint:
- **Mobile (< lg)**: Slide-in overlay panel (85vw width), toggle button visible
- **Desktop (lg+)**: Fixed position panel, always visible, toggle hidden

## Safe Area Support

The layout handles modern device safe areas (notches, home indicators):

```css
/* Bottom navigation */
padding-bottom: calc(64px + env(safe-area-inset-bottom, 0px));

/* Side content */
padding-left: env(safe-area-inset-left, 0px);
padding-right: env(safe-area-inset-right, 0px);
```

## Mobile-Specific Design Decisions

1. **Simplified navigation** - Bottom nav with 5 items vs desktop sidebar with all sections
2. **Reduced information density** - Fewer columns, larger cards, less data per view
3. **Sheet-based modals** - Bottom sheet patterns instead of centered dialogs
4. **Stacked layouts** - Content organized vertically instead of side-by-side
5. **Contextual sidebar as overlay** - Doesn't consume screen real estate when closed
6. **No hover states** - Touch-based interaction patterns replace hover effects
